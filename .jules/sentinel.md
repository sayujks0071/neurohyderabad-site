## 2024-05-15 - [API Key Verification Timing Attack]
**Vulnerability:** API key verification used standard string equality (`===`) instead of `crypto.timingSafeEqual`, making the endpoints vulnerable to timing attacks.
**Learning:** Found multiple instances of `apiKey === validKey` in `app/api/workflows/*/route.ts`. Standard string comparison exits early on the first mismatched character, allowing an attacker to guess the API key character by character by measuring the response time.
**Prevention:** Always use `crypto.timingSafeEqual` after hashing both the input and expected keys with SHA-256 for secure comparison.

## 2024-05-24 - [HIGH] crypto.timingSafeEqual Buffer Length DoS
**Vulnerability:** The application used `crypto.timingSafeEqual` to compare a user-supplied string (`hash`) with a computed string (`generatedHash`) after wrapping both in `Buffer.from()`. If an attacker provided a malformed signature that produced a `hash` of a different length than the expected HMAC hex string, `Buffer.from()` would create buffers of differing lengths. `crypto.timingSafeEqual` strictly throws a `TypeError: Input buffers must have the same byte length` if the input buffers are unequal. This unhandled exception would crash the Node.js process, leading to a Denial of Service (DoS) attack via malicious webhooks.
**Learning:** `crypto.timingSafeEqual` cannot be used directly on unhashed user input if there's any possibility the lengths might differ. It is not sufficient to just compare buffers; they must be guaranteed to be the exact same length before the function is called.
**Prevention:** Always hash both the user-provided input and the expected secret using a strong hashing algorithm (like `crypto.createHash('sha256')`) *before* passing them to `crypto.timingSafeEqual`. This guarantees that both inputs will be exactly 32 bytes (256 bits) long, preventing length-mismatch exceptions while still providing constant-time comparison against timing attacks.
